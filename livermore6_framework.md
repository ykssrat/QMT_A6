# 利弗莫尔 6.0 框架蓝图（草案）

## 0. 目标
本文档用于定义利弗莫尔 6.0 的第一版架构蓝图。
目标：将当前偏探索式流程，升级为一致、可进化、可验证的研究系统。

核心原则：
- 策略优先：保留利弗莫尔投资哲学（截断亏损，让利润奔跑）。
- 进化优先：使用可进化优化器（GA 或兼容算法）优化策略参数。
- 可靠性优先：强制接口契约、可复现实验、数据与模式校验。
- 跨资产可扩展：同一工作流同时支持基金与股票。

---

## 1. 优化系统：可进化算法 + 三类目标方向

### 1.1 可进化优化器选型
主选方案：
- 遗传算法（GA）：适合规则驱动、目标面不光滑的策略优化问题。

6.x 扩展路线：
- CMA-ES：连续参数空间下的高效搜索。
- NSGA-II：真正多目标的 Pareto 优化。
- 混合方案：GA 全局搜索 + 局部精修。

### 1.2 三类目标方向（必选）
按进化目标定义三种 profile：

1. 收益导向（`target=return`）
- 最大化累计收益。
- 保留基本风控护栏，避免“暴雷型”虚高解。

2. 稳定导向（`target=stability`）
- 最大化风险调整后质量（如 Calmar 类指标）。
- 强化回撤、换手等约束。

3. 平衡导向（`target=balanced`）
- 收益与稳定性加权融合。
- 权重必须配置化，禁止硬编码。

### 1.3 统一评分契约
每个候选参数评估后必须输出标准化指标：
- `return_pct`
- `max_drawdown_pct`
- `calmar_like`
- `turnover`
- `trade_count`
- `cash_utilization`
- `fitness_raw`
- `fitness_final`

意义：
- 优化器与策略内部实现解耦。
- 报告与冠军榜评估口径可重复、可追踪。

---

## 2. 保留 MCAH 策略核心 + 参数半衰期路线

### 2.1 保留 MCAH 作为策略骨架
必须保留的行为内核：
- `m`：试探建仓系数（低风险启动）。
- `c`：止损边界（硬性截断亏损）。
- `a`：盈利确认后的加仓攻击强度。
- `h`：盈利解锁阈值（确认优势后再加仓）。

利弗莫尔哲学保持不变：
- 通过 `c` 快速截断错误。
- 通过 `h + a` 让利润段扩张。

### 2.2 参数域治理
- 所有物理参数区间必须配置化。
- raw <-> physical 映射必须唯一真源。
- 禁止训练引擎/回测引擎/绘图/报告各自维护映射公式。

### 2.3 参数半衰期估计路线
沿用 EEW 思想，但协议更严格：
1. 围绕精英参数对每个维度进行受控微扰。
2. 观察日收益与动作决策的滚动影响。
3. 使用鲁棒回归拟合衰减（降低离群点影响）。
4. 输出：
- 参数级半衰期估计；
- 置信区间；
- 市场状态标签（趋势/震荡/高波动）。

目标解释层：
- 不只回答“能活多久”，还要回答“在哪种市场状态下失效最快”。

## 3. 实力胜率问题：面向未来的设计

### 3.1 为什么单次切分不够
单一 train/test 切分路径依赖强，容易把“幸运行情命中”误判为“真实能力”。

### 3.2 实力胜率定义（6.0）
对单一候选策略：
- 在 K 个场景评估（时间窗 + 切分种子 + 可选资产子集）。
- 每个场景与基线策略对比。
- `skill_winrate = wins / K`。

### 3.3 融合方式
- 保留目标 profile 的基础 fitness。
- 用可配置系数融合 `skill_winrate`。
- 工件中同时保存（`fitness_base`, `skill_winrate`, `fitness_final`）。

### 3.4 场景生成建议
为了降低“小基金池 + 高重合”带来的伪稳健：
- 时间轴切片：滚动窗 + 锚定窗。
- 资产池扰动：按板块/基金/股票采样，并设置重合约束。
- 状态覆盖：每次实验强制覆盖多种波动状态。

## 4. 必须遵守的项目结构（6.0）

顶层目录固定为：
- `/docs`
- `/fund`
- `/stock`
- `/config`
- `/output`
- `/data`

建议结构：

```text
/docs/
  livermore6_framework.md

/stock/
  scripts/
```

- `fund` 与 `stock` 模块保持独立。
- 但两者必须遵循同一接口契约，便于复用优化/报告链路。

## 5. 数据与回测要求

### 5.1 用户自定义标的池
系统必须支持用户自定义基金或股票代码列表。

输入方式：
- `/config/data/` 中的 YAML/JSON；
- CLI 菜单选择；
- 运行时传入临时代码列表覆盖。

### 5.2 时间尺度支持
- 回测时间尺度可选 1~10 年。
- 同时支持锚定窗与滚动窗。

### 5.3 新基金容忍（如仅 2025-09 后成立）
短历史标的处理规则：
- 不因单个标的历史不足导致全局实验失败。
- 使用 warm-up fallback：
  - 历史不足时使用可用短窗；
  - 输出可靠性标记；
  - 必要时从长窗统计中剔除该标的。
- 未来可选 proxy 模式：成立前使用板块指数代理。

### 5.4 数据质量闸门（强制）
训练前必须执行：
- 字段模式校验（date/close/volume/可用性标记）；
- 数值字段强制类型转换，拒绝非法 dtype；
- 输出每个标的的缺失率和可用历史长度；
- 仅在“最小可用样本阈值”不达标时整体中止。

## 6. 接口契约（6.0 稳定性的关键）

### 6.1 策略契约
`strategy.step(state, params) -> action_plan`

### 6.2 环境契约
`env.run_with_params(params) -> metrics_dict`

### 6.3 优化器契约
`optimizer.search(evaluator) -> best_params, run_summary`

### 6.4 报告器契约
`reporter.build(run_artifact_dir) -> pdf_path`

意义：
- 避免接口漂移和模块乱耦合。
- 优化器可替换（如从 GA 切换 NSGA-II）而无需重写策略引擎。

---

## 7. 实验与可复现协议

每次运行必须产出的工件：
- 配置快照（所有生效参数）；
- 随机种子（切分种子 + 优化种子）；
- 数据快照 hash/时间戳；
- 完整指标与交易日志。

建议运行模式：
- `debug`：小种群、短时间窗。
- `research`：标准研究模式。
- `benchmark`：固定协议用于版本横向对比。

---

## 8. 从 5.0 到 6.0 的迁移建议

A 阶段（契约稳定化）：
- 冻结 5.0 行为。
- 抽离共享接口与 schema 校验器。

B 阶段（核心引擎清理）：
- 统一参数映射真源。
- 训练/评估/报告完全拆模块。

C 阶段（高级进化）：
- 引入场景生成器。
- 引入实力胜率融合目标。
- 引入状态感知半衰期分析。

D 阶段（跨资产整合）：
- 股票路径对齐基金路径接口。
- 统一实验治理与报告体系。

---

## 9. 附录：5.0 已暴露失败模式与 6.0 防线

已观察失败类型：
- 市场序列 dtype 漂移导致运行期深层崩溃。

6.0 强制防线：
- 训练前严格做类型清洗，非数值列提前拒绝。
- 不允许把 dtype 问题拖到训练循环深处才暴露。

该规则可以显著降低“长时间计算后才崩溃”的损失。
